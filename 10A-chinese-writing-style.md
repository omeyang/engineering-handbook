# 中文技术写作规范

**上级文档**：[10-documentation-standards.md](./10-documentation-standards.md)

本附录定义中文技术文档的核心书写规范，聚焦**技术文档高频使用**的规则。

---

## 1. 中英文混排规范（必须遵守）

### 1.1 空格使用规则

**核心原则**：中英文之间**必须**加空格，提高可读性。

| 场景 | 规则 | 错误示例 | 正确示例 |
|------|------|---------|---------|
| 中文与英文 | **必须**加空格 | `使用MongoDB数据库` | `使用 MongoDB 数据库` |
| 中文与数字 | **必须**加空格 | `共100个文件` | `共 100 个文件` |
| 中文与百分号 | **禁止**加空格 | `覆盖率 50 %` | `覆盖率 50%` |
| 数字与单位 | **禁止**加空格 | `100 GB` | `100GB` |
| 中文与代码 | **必须**加空格 | 使用`NewCache()`函数 | 使用 `NewCache()` 函数 |

**典型错误示例**：
```markdown
❌ 服务通过gRPC接口提供查询功能，QPS达到8000次/秒。
✅ 服务通过 gRPC 接口提供查询功能，QPS 达到 8000 次/秒。
```

## 2. 标点符号规范（技术文档核心规则）

### 2.1 中英文标点选择

**核心原则**：中文内容用中文标点，英文内容用英文标点。

| 标点 | 中文 | 英文 | 使用场景 |
|------|------|------|---------|
| 句号 | `。` | `.` | 中文句子用`。` |
| 逗号 | `，` | `,` | 中文句子用`，` |
| 顿号 | `、` | 无 | 中文并列词语（英文用`,`） |
| 冒号 | `：` | `:` | 中文句子用`：` |

**示例**：
```markdown
❌ 服务包含以下模块: 资产管理, 归属判定, 状态匹配.
✅ 服务包含以下模块：资产管理、归属判定、状态匹配。
```

---

### 2.2 顿号使用（技术文档最常见的 3 条规则）

**规则 1：并列词语用顿号，不用逗号**
```markdown
❌ TiDB，Pump，TiKV，PD 是核心组件。
✅ TiDB、Pump、TiKV、PD 是核心组件。
```

**规则 2：连词前不用顿号**
```markdown
❌ 系统支持 MySQL、PostgreSQL、以及 MongoDB。
✅ 系统支持 MySQL、PostgreSQL 以及 MongoDB。
```

**规则 3：阿拉伯数字序号后用点号，不用顿号**
```markdown
❌ 1、资产管理  2、归属判定
✅ 1. 资产管理  2. 归属判定
```

---

### 2.3 连接号规范（技术文档最常见的 3 种）

**核心原则**：三种连接号用途不同，**禁止**混用。

| 符号 | 名称 | 用途 | 示例 |
|------|------|------|------|
| **—** | 一字线 | 时间起止、空间起止 | `2024 年 1 月—6 月`、`北京—上海` |
| **-** | 短横线 | 复合名词、日期、图表编号 | `asset-enrichment`、`2024-01-18`、`图 2-1` |
| **～** | 浪纹线 | 数值范围 | `1～10 GB`、`8000～12000 QPS` |

**记忆口诀**：
- **起止用一字线**（—）：时间、空间的开始到结束
- **范围用浪纹线**（～）：数值的波动范围
- **复合用短横线**（-）：多个词组合成一个概念

**常见错误**：
```markdown
❌ 温度范围：-36--8℃（错误：应用浪纹线）
✅ 温度范围：-36～-8 ℃

❌ 2024年1月～6月（错误：应用一字线）
✅ 2024 年 1 月—6 月
```

---

### 2.4 括号规范（技术文档核心规则）

**核心原则**：根据括号内容选择括号形式和空格规则。

| 括号内容 | 括号类型 | 空格规则 | 示例 |
|---------|---------|---------|------|
| 全为英文 | 半角 `()` | 前后各空一格 | 数据定义语言 (DDL) 是…… |
| 中英混合 | 全角 `（）` | 前后不空格 | 斜杠（slash 或 forward slash）用于…… |
| 全为中文 | 全角 `（）` | 前后不空格 | 资产管理（包括创建、更新、删除）…… |

**常见错误**：
```markdown
❌ 数据定义语言（DDL）是……（错误：全英文应用半角括号）
✅ 数据定义语言 (DDL) 是……

❌ 斜杠 (slash 或 forward slash) 用于……（错误：中英混合应用全角括号）
✅ 斜杠（slash 或 forward slash）用于……
```

---

### 2.5 列表标点规范

**核心原则**：列表项末尾标点**必须**保持一致。

**选项 1：完整句子 -> 每项末尾加句号**
```markdown
✅ 正确：
- 管理模块负责资源的创建、更新和删除。
- 判定模块负责组织归属匹配。
```

**选项 2：短语 -> 每项末尾不加标点**
```markdown
✅ 正确：
- 管理模块
- 判定模块
```

**错误：混合使用**
```markdown
❌ 错误：
- 管理模块负责资源的创建、更新和删除。
- 判定模块
```

## 3. 数字使用规范（技术文档核心原则）

### 3.1 数字表达选择

**核心原则**：技术文档**优先**使用阿拉伯数字，保证精确性。

| 场景 | 规则 | 示例 |
|------|------|------|
| 统计数据 | **必须**用阿拉伯数字 | `8000 QPS`、`2.1GB`、`85ms` |
| 大数字 | **应该**用"万/亿"简化 | `125 万条记录`、`8000 QPS` |
| 序号编号 | **必须**用阿拉伯数字 | `原则 1`、`第 3 章` |

**示例对比**：
```markdown
❌ 系统包含一千二百五十万条记录，QPS达到八千。
✅ 系统包含 1250 万条记录，QPS 达到 8000。
```

**注意**：技术文档**禁止**使用概数（如"三五天"、"七八十个"），**必须**用精确数字。

---

### 3.2 数字区间表达

**核心原则**：使用连接号表示数字区间（参见 2.3 连接号规范）。

| 场景 | 连接号 | 示例 |
|------|--------|------|
| 数值范围 | 浪纹线 `～` | `1～10 GB`、`8000～12000 QPS` |
| 时间起止 | 一字线 `—` | `2025 年 1 月—6 月` |

## 4. 专有名词规范

### 4.1 大小写规则

**核心原则**：**必须**保持专有名词的官方大小写。

**常见专有名词表**：

| 类别 | 正确写法 | 错误写法 |
|------|---------|---------|
| **数据库** | MongoDB、MySQL、PostgreSQL、Redis | Mongodb、mysql、postgresql |
| **消息队列** | Kafka、Pulsar、RabbitMQ | kafka、pulsar |
| **容器技术** | Docker、Kubernetes、Podman | docker、kubernetes |
| **编程语言** | Go、Python、JavaScript、TypeScript | Golang、python、Javascript |
| **协议** | HTTP、HTTPS、gRPC、WebSocket | Http、grpc、websocket |
| **版本控制** | Git、GitHub、GitLab | git、github、gitlab |

**注意**：Go 语言的官方名称是 **Go**，不是 **Golang**。

## 5. 常见错误（技术文档禁止）

### 5.1 口语化表达（必须删除）

**核心原则**：技术文档**禁止**使用口语化、不确定性表达。

| 错误表达 | 问题 | 正确表达 |
|---------|------|---------|
| 可能、大概、也许、应该 | 不确定性 | 使用确定性表述或标注"待确认" |
| 一般来说、通常情况下 | 模糊 | 明确具体场景或条件 |
| 比较快、相对稳定 | 主观 | 使用具体数据（如"P95 延迟 85ms"） |
| 差不多、基本上 | 不精确 | 使用精确数据或范围 |

**示例对比**：
```markdown
❌ 系统性能可能会比较好，一般来说 QPS 大概能达到几千左右。
✅ 系统性能测试结果：QPS 达到 8000，P95 延迟 85ms。
```

---

### 5.2 冗余表达（应该简化）

**核心原则**：删除不必要的修饰词和冗余表达。

| 冗余表达 | 简化表达 |
|---------|---------|
| 进行部署 | 部署 |
| 进行优化 | 优化 |
| 进行测试 | 测试 |
| 做出决策 | 决策 |

**示例对比**：
```markdown
❌ 对系统进行部署后，需要进行性能测试。
✅ 系统部署后，需要性能测试。
```

---

### 5.3 歧义表达（必须消除）

**核心原则**：避免产生歧义的表达。

**数量范围歧义**：
```markdown
❌ 部署 3-5 个服务器（歧义：3 到 5 个？还是 3 减 5？）
✅ 部署 3～5 个服务器（明确：范围）
```

**修饰对象歧义**：
```markdown
❌ 快速的 MongoDB 数据库查询（歧义：MongoDB 快？还是查询快？）
✅ 快速查询 MongoDB 数据库
```

**并列关系歧义**：
```markdown
❌ 用户管理和权限校验模块（歧义：一个模块还是两个？）
✅ 用户管理模块和权限校验模块（明确：两个模块）
```

## 6. 链接文字规范

### 6.1 链接文字选择

**核心原则**：**禁止**使用"点击这里"、"请看这里"等无意义文字。

```markdown
❌ 详细信息请看[这里](./architecture.md)。
✅ 详见 [架构设计文档](./architecture.md)。
```

### 6.2 代码位置引用

**核心原则**：代码位置引用**必须**使用反引号格式。

```markdown
❌ 详见 internal/service/handler.go:123-156
✅ 详见 `internal/service/handler.go:123-156`
```

## 7. 术语一致性规范

### 7.1 术语统一原则

**核心原则**：同一概念在文档中**必须**使用统一术语。

**示例**：

```markdown
❌ 错误（同一概念多种表述）：
- 认证模块使用 JWT 校验。
- Auth 模块负责身份验证。
- 鉴权功能通过中间件实现。

✅ 正确（统一术语）：
- 认证模块使用 JWT 校验。
- 认证模块负责身份验证。
- 认证模块通过中间件实现。
```

### 7.2 中英文对照

**首次出现时标注英文**：

```markdown
✅ 正确：
认证（Authentication）模块负责身份验证。

后续使用中文即可：
认证模块支持多种验证方式。
```

## 8. 检查清单

### 中英文混排检查

- [ ] 中文与英文之间加空格
- [ ] 中文与数字之间加空格
- [ ] 百分号和单位不加空格
- [ ] 代码引用前后加空格

### 标点符号检查

- [ ] 中文内容使用中文标点
- [ ] 英文内容使用英文标点
- [ ] 并列词语使用顿号，连词前不用顿号
- [ ] 连接号根据用途选择：一字线/短横线/浪纹线
- [ ] 括号根据内容选择：全英文用半角 `()`，中英混合用全角 `（）`
- [ ] 列表项标点保持一致

### 数字使用检查

- [ ] 统计数据使用阿拉伯数字
- [ ] 禁止使用概数，必须用精确数字
- [ ] 数字区间：数值范围用浪纹线 `～`，时间起止用一字线 `—`

### 专有名词检查

- [ ] 保持官方大小写（GitHub、MongoDB、API）
- [ ] 术语在文档中保持一致
- [ ] 首次出现标注英文

### 常见错误检查

- [ ] 删除口语化表达（可能、大概、也许、应该、基本上、差不多）
- [ ] 简化冗余表达（进行部署 -> 部署）
- [ ] 消除歧义表达（数量范围、修饰对象、并列关系）

### 链接文字检查

- [ ] 不使用"点击这里"、"请看这里"
- [ ] 链接文字描述目标内容
- [ ] 代码位置使用反引号格式

## 9. 自动化检查工具（推荐）

虽然本规范**必须**手动遵守，但**可以**使用以下工具辅助检查：

| 工具 | 检查项 | 说明 |
|------|--------|------|
| `autocorrect` | 中英文混排、标点符号 | 全面的中文技术文档检查工具 |
| `pangu` | 中英文空格 | 自动在中英文之间加空格 |
| 编辑器插件 | 实时提示 | VS Code、Vim 等都有相关插件 |

**注意**：自动化工具仅作为辅助，最终**必须**人工审查确认。
